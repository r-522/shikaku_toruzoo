# 02 基本設計書

## 1. システム構成

### 1.1 全体アーキテクチャ

```
┌──────────────────────────────────────────────────────────────────┐
│                        Google Cloud Run                          │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │                shikaku_toruzoo コンテナ                       │ │
│  │                                                             │ │
│  │  ┌─────────────────────────────────────────────────────┐   │ │
│  │  │  Actix-web (Rust)                                    │   │ │
│  │  │  ・API エンドポイント (/api/*)                         │   │ │
│  │  │  ・静的ファイル配信 (Vue ビルド成果物 → /app/static)    │   │ │
│  │  │  ・SPA フォールバック (非/api/* → index.html)          │   │ │
│  │  └─────────────────────────────────────────────────────┘   │ │
│  └─────────────────────────────────────────────────────────────┘ │
│                             │ HTTPS (マネージド証明書)              │
└─────────────────────────────┼────────────────────────────────────┘
                              ▼
                  ┌──────────────────────┐
                  │     Supabase          │
                  │   (PostgreSQL)        │
                  │  PostgREST API経由    │
                  └──────────────────────┘
```

### 1.2 技術選定

| 技術 | 選定理由 |
|------|---------|
| Rust (Actix-web 4) | 型安全性・メモリ安全性・高パフォーマンス。低リソース消費でCloud Runに最適 |
| Vue 3 + TypeScript | Composition APIによるロジック再利用性。TypeScriptで型安全なフロントエンド |
| Bootstrap 5 (CSSのみ) | レスポンシブ対応。JSバンドルは使用せず、インタラクションはVueで実装 |
| Pinia | Vue 3公式の状態管理ライブラリ。シンプルで型安全なストア設計 |
| Supabase (PostgreSQL) | PostgreSQLベースのBaaS。PostgREST経由でRustから直接REST APIアクセス |
| Google Cloud Run | コンテナベースのサーバーレス。オートスケーリングとHTTPS自動対応 |
| Argon2id | OWASP推奨パスワードハッシュ。サイドチャネル攻撃耐性 |
| HMAC-SHA256 | メールアドレスの検索可能なハッシュ化。サーバー管理キーによる安全性 |

## 2. 画面設計

### 2.1 画面一覧

| 画面ID | 画面名 | パス | 認証 | 概要 |
|--------|--------|------|------|------|
| SCR-001 | サインアップ | /signup | 不要 | 新規ユーザー登録 |
| SCR-002 | サインイン | /signin | 不要 | ログイン |
| SCR-003 | ダッシュボード | /dashboard | 必要 | 所持資格・目標・コミュニティを統合した単一画面 |
| SCR-004 | ユーザー詳細 | /community/:id | 必要 | 特定ユーザーの資格・目標詳細 |

> 旧設計の /certifications, /goals, /community は /dashboard にリダイレクト。

### 2.2 画面遷移

```
[サインアップ] ──登録完了──▶ [サインイン]
                                  │
                            認証成功
                                  ▼
                          [ダッシュボード]
                         （所持資格 / 目標 / コミュニティ）
                                  │
                         ユーザーカードクリック
                                  ▼
                          [ユーザー詳細]
```

### 2.3 画面レイアウト方針

#### SCR-001 / SCR-002: サインアップ / サインイン
- 画面中央にフォームカード1枚。背景: #F5F5F5
- プライマリカラー #1A73E8 の単一ボタン
- バリデーションエラーはフィールド直下に赤テキスト (#D32F2F) で即時表示

#### SCR-003: ダッシュボード
- ヘッダー: アプリ名「資格取るぞー！」+ ユーザー名 + サインアウトボタン
- サマリーカード: 所持資格数 / 合格数 / 合計勉強時間
- 所持資格セクション: カード形式CRUD。新規登録ボタン（ドロップダウン切り替え）
- 目標セクション: ステータス別（受験日/合格/不合格/断念）でグループ表示。勉強時間+/-ボタン
- コミュニティセクション: 他ユーザーのカード一覧（お気に入り優先）。所持資格・目標・勉強時間を表示

#### SCR-004: ユーザー詳細
- ユーザー名・グッドマークを上部表示
- 所持資格リスト・目標リストを表示（閲覧のみ）

## 3. API設計

### 3.1 エンドポイント一覧

| Method | Path | 認証 | 概要 |
|--------|------|------|------|
| POST | /api/auth/signup | 不要 | ユーザー登録 |
| POST | /api/auth/signin | 不要 | サインイン |
| POST | /api/auth/signout | 必要 | サインアウト |
| GET | /api/auth/me | 必要 | 認証ユーザー情報取得 |
| GET | /api/certifications | 必要 | 所持資格一覧 |
| POST | /api/certifications | 必要 | 所持資格登録 |
| PUT | /api/certifications/:id | 必要 | 所持資格更新 |
| DELETE | /api/certifications/:id | 必要 | 所持資格削除 |
| GET | /api/goals | 必要 | 目標一覧 |
| POST | /api/goals | 必要 | 目標登録 |
| PUT | /api/goals/:id | 必要 | 目標更新 |
| DELETE | /api/goals/:id | 必要 | 目標削除 |
| GET | /api/master/search?q= | 必要 | 資格マスタ検索 |
| GET | /api/community/users | 必要 | コミュニティユーザー一覧 |
| GET | /api/community/users/:id | 必要 | コミュニティユーザー詳細 |
| POST | /api/favorites | 必要 | お気に入り登録 |
| DELETE | /api/favorites/:target_id | 必要 | お気に入り解除 |

### 3.2 認証方式

- Cookie: `session_token` (HttpOnly / Secure / SameSite=Strict)
- セッション有効期限: 1年間（サインアウトまで保持）
- 認証ミドルウェア: Cookie → TBL_SESSION照合 → AuthenticatedUser抽出
- レート制限: 認証エンドポイント 5回/分、その他 60回/分

## 4. データベース設計

### 4.1 ER図（概要）

```
TBL_USER ──┬── TBL_SESSION
           ├── TBL_HOLDING ── TBL_MASTER
           ├── TBL_GOAL    ── TBL_MASTER
           └── TBL_FAVORITE (自己参照)
```

### 4.2 テーブル一覧

| テーブル名 | 概要 |
|-----------|------|
| TBL_USER | ユーザー（パスワード・メールアドレスはハッシュ化済み） |
| TBL_SESSION | セッショントークン管理 |
| TBL_MASTER | 資格名マスタ（全ユーザー共通） |
| TBL_HOLDING | 所持資格（ユーザー×マスタ） |
| TBL_GOAL | 目標（ユーザー×マスタ、勉強時間含む） |
| TBL_FAVORITE | お気に入り（ユーザー×対象ユーザー） |

### 4.3 主要カラム

**TBL_USER**

| カラム | 型 | 概要 |
|-------|-----|------|
| useid | UUID PK | ユーザーID |
| usenm | VARCHAR(30) | ユーザー名 |
| useml | VARCHAR(128) UNIQUE | メールアドレス（HMAC-SHA256ハッシュ） |
| usepw | VARCHAR(256) | パスワード（Argon2idハッシュ） |
| useca | TIMESTAMPTZ | 作成日時 |
| useua | TIMESTAMPTZ | 更新日時 |

**TBL_GOAL**

| カラム | 型 | 概要 |
|-------|-----|------|
| goaid | UUID PK | 目標ID |
| goaui | UUID FK | ユーザーID |
| goami | UUID FK | マスタID |
| goatd | DATE | 目標日 |
| goast | VARCHAR(20) | ステータス（exam_date/passed/failed/abandoned） |
| goamm | TEXT | メモ（最大1000文字） |
| goash | NUMERIC(6,1) | 勉強時間（0.5h単位） |
| goaca | TIMESTAMPTZ | 作成日時 |
| goaua | TIMESTAMPTZ | 更新日時 |

## 5. セキュリティ設計

| 項目 | 方針 |
|------|------|
| パスワード | Argon2id (メモリ: 65536KB, 反復: 3, 並列: 4) |
| メールアドレス | HMAC-SHA256 + サーバー管理シークレット（正規化後ハッシュ） |
| セッション | 256bit暗号学的乱数トークン → Base64エンコード |
| Cookie | HttpOnly / Secure / SameSite=Strict |
| CORS | CORS_ORIGIN環境変数で指定したオリジンのみ許可 |
| CSRF対策 | SameSite=Strict Cookie + X-Requested-With ヘッダー検証 |
| ソースマップ | 本番ビルドで無効化 (vite.config.ts) |
